# ZimRate — Changes of 2026-06-10 (security hardening + UI/admin sweep)

Reference for agents working on or with this codebase (blog pipeline, monitors, future dev sessions). Everything below shipped in one sweep on 2026-06-10. All 51 tests pass; every public and admin page was smoke-tested against a running instance.

---

## 1. Impact on pipeline agents — read this first

- **`/api/blog` is unchanged.** Posting, patching, publishing via `X-Admin-Token` works exactly as before. No pipeline changes required.
- **`X-Admin-Token` auth is unchanged server-side**, but the token VALUE must be treated as compromised: it was previously hardcoded in two admin templates and is in git history. **After Tino rotates `ADMIN_TOKEN` in production, agents must use the new value.**
- **New rate limits (per IP, fixed 15-minute window, HTTP 429 when exceeded):**
  - `POST /admin/login` — 10 per window
  - `POST /api/alerts/subscribe` and `POST /api/newsletter/subscribe` — 8 per window
  - Nothing else is limited. If a monitor/test agent probes the subscribe endpoints repeatedly, expect 429 with body `{"error":"Too many requests. Please try again later."}`.
- **`/admin/blog` (HTML) is now paginated** (20/page, `?page=N`). Agents should not scrape admin HTML anyway — use `GET /api/blog?status=...`.
- **New public page: `GET /api-docs`** — developer API documentation. It is in the sitemap. Browser smoke/deep-check monitors should add it to their page list.
- **New audit event type `MANUAL_RATE`** can now appear in `GET /api/admin/events` (logged when a rate is entered manually in the admin).
- **Template convention:** the rate-alert modal is now a shared fragment `layout :: alert-modal`. Any new public template must include `<th:block th:replace="~{layout :: alert-modal}"></th:block>` before the footer include, or the header bell button silently does nothing on that page.
- **Cache-busting:** public templates reference `styles.css?v=20260610-ui1`. When changing styles.css, bump this param in all public templates.

## 3. Public (user-facing) UI

**Shared alert modal** — markup + JS extracted from `index.html` into `th:fragment="alert-modal"` in `templates/layout.html`, now included on every public page (index, convert, history, blog-list, blog-post, about, contact, privacy, widget, remittance, api-docs). The header bell works site-wide. Behavior added:

- Escape key closes the modal; first input is focused on open; focus returns to the opener on close.
- Modal tabs have `role="tablist"` / `aria-selected`; JS keeps them in sync.

**Forms** — alert-subscribe and newsletter forms (blog-list, blog-post) now show a user-facing error ("Something went wrong — please try again.") on failed/non-2xx requests, and success messages auto-dismiss after 4 seconds.

**Accessibility (`index.html`)** — filter pills got `aria-pressed` + labels (JS-maintained); converter swap element got an aria-label; "best rate" rows carry a visually-hidden "Best rate in this section" span (no longer color-only).

**styles.css**

- `.ad-slot--leaderboard` / `--rectangle` reserve min-heights (90px / 280px; 60px mobile) so AdSense ads won't cause layout shift.
- `.section-sub` at ≤480px: smaller font (0.68rem) instead of `display: none` (content no longer hidden on phones).
- Dark-mode contrast: all 24 foreground uses of `#64748b` in `[data-theme="dark"]` rules bumped to `#94a3b8` (WCAG AA).
- New `.history-table-head` / `.history-csv-btn` styles; new appended, clearly-commented "API DOCS PAGE" section at end of file (`.api-docs-*`, `.api-code`, `.api-params-table`, etc.) with dark variants.

**history.html** — "Download CSV" button beside the daily-rates table; client-side CSV generation, file name `zimrate-history-<monthSlug>.csv` (e.g. `zimrate-history-june-2026.csv`). Only rendered when rows exist.

**layout.html footer** — added links: API (`/api-docs`) and Widget (`/widget`).

Note: ApexCharts loading was audited — it is already only included on the three pages that render charts (index, convert, history). No change was needed.

Known untestable gaps (hardcoded constants, no injectable clock): rate-limit window expiry, Jsoup scraper retry timing.
